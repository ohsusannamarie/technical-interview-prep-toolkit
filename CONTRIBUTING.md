# Contributing

Thanks for helping improve this toolkit.

This repo is meant to stay practical, candidate-friendly, and easy for recruiters to share. It is not trying to become the largest technical interview prep list on GitHub.

## Contribution philosophy

Quality beats quantity.

A resource should earn its place because it helps someone prepare with more clarity, not because it makes the repo look bigger.

## Good contributions

Helpful contributions include:

- Candidate-friendly resources
- Clear coding practice links
- System design resources
- Behavioral interview prep resources
- Role-specific prep guidance
- Recruiter enablement language
- Interview checklists
- Accessibility improvements
- Broken link fixes
- Better organization or clearer explanations

## Resource criteria

Before suggesting a resource, ask:

- Is it useful for real candidates?
- Is it clear what skill it helps build?
- Is it free or reasonably accessible?
- Is it specific enough to be helpful?
- Would a recruiter feel comfortable sending it to a candidate?
- Does it reduce ambiguity instead of adding noise?
- Is it current enough to trust?

## What to avoid

Please avoid:

- Spammy links
- Affiliate-heavy resources
- Giant link dumps with no context
- Resources that require expensive subscriptions without noting that clearly
- Content that is outdated, misleading, or overly generic
- Advice that encourages candidates to memorize answers instead of understanding concepts
- Fear-based advice that makes candidates feel like they are already behind

## Suggested format for new resources

When adding a resource, use this format:

```md
### Resource Name

https://example.com

Best for:

- Skill or use case
- Skill or use case
- Skill or use case

Notes:

- Add any relevant context, such as beginner-friendly, advanced, free, paid, or best used for a specific interview type.
```

## Suggested format for role prep

When adding or editing role-specific prep, use this structure:

```md
## Role Name

Focus areas:

- Skill area
- Skill area
- Skill area

Prepare to discuss:

- Example discussion topic
- Example discussion topic

Good prep:

- Practical prep action
- Practical prep action

Strong signal:

- What a strong candidate demonstrates in this interview.
```

## Tone

Keep the tone:

- Clear
- Practical
- Human
- Encouraging
- Not robotic
- Not fear-based

Candidates are already carrying enough interview stress. The repo should help them feel more oriented, not more behind.

## Pull requests

For pull requests:

1. Keep changes focused
2. Explain what you changed and why
3. Include context for new resources
4. Check links before submitting
5. Avoid adding resources just to make the list longer
6. Use descriptive commit messages when possible

## Good pull request examples

Strong PRs might include:

- Adding a high-quality system design resource with notes on when to use it
- Fixing broken links
- Improving a confusing section
- Adding prep guidance for a missing role type
- Making a checklist more candidate-friendly

## Not-so-great pull request examples

Less helpful PRs might include:

- Adding 20 unrelated links with no explanation
- Adding paid resources without noting cost
- Adding generic advice that could apply to any interview anywhere
- Rewriting the tone into corporate fog machine language

## Final note

The goal is simple: help candidates and recruiters reduce ambiguity.

If a change makes the repo clearer, calmer, and more useful, it probably belongs here.
